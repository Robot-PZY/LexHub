# 律枢 LexHub 国赛答辩 PPT 方案

团队名称：六百六十六  
项目名称：律枢 LexHub  
项目定位：基于中兴 Co-Sight 的法律业务智能服务平台

## 1. 答辩 PPT 总体风格

本 PPT 面向国赛现场答辩、线上评审展示和演示视频剪辑。项目正式全称为“律枢 LexHub：基于中兴 Co-Sight 的法律业务智能服务平台”。整体采用竞赛答辩中常见的“短标题、强图示、少段落、重证据”风格：每页只讲一个问题，用图表、截图、指标和代码/配置片段支撑观点。

不建议做成论文式长段文字，也不建议做成产品发布会式大面积炫彩页面。LexHub 是法律业务场景下的工程项目，视觉上应保持克制、清晰、可信，突出“法律任务链路、Co-Sight 改造、多智能体协作、可复核过程、商业落地价值”。

推荐版式规格：

| 项目 | 建议 |
|---|---|
| 页面比例 | 16:9 宽屏 |
| 主色 | 深蓝、石墨灰、白色 |
| 强调色 | 少量琥珀色或青蓝色，用于编号、箭头、关键指标 |
| 字体层级 | 标题 32-40pt，小标题 20-24pt，正文 16-18pt，注释 11-12pt |
| 页面密度 | 每页 1 个主图或 1 个主表，最多 3 个核心结论 |
| 图表风格 | 白底、细线框、浅灰分区、少量强调色 |
| 截图使用 | 控制在 4 至 5 页出现，优先展示真实运行闭环 |

通用页面结构：

```text
┌────────────────────────────────────────────────────────────┐
│ 页面标题：短词组，如“业务背景与痛点”                         │
│ 副标题：一句话说明本页观点                                   │
├────────────────────────────────────────────────────────────┤
│ 主视觉区：架构图 / 流程图 / 截图 / 对比表 / 指标图            │
├──────────────────────────────┬─────────────────────────────┤
│ 关键证据：指标、代码、配置    │ 讲解要点：3 条以内            │
└──────────────────────────────┴─────────────────────────────┘
```

## 2. 定稿页码结构

定稿采用 **15 页固定结构**。不再设置备选页或备份页；每页均可进入正式 PPT。若答辩时间较短，可压缩讲解时间，但不调整页码。

| 页码 | 标题 | 叙事任务 | 页面类型 |

| 7 | 智能体角色分工 | 展示法律智能体组织方式 | 角色分工页 |

| 9 | 功能演示：任务执行 | 展示任务提交、材料上传、DAG 执行 | 截图演示页 |
| 10 | 功能演示：结果交付 | 展示结果页、文书导出、Replay 回放 | 截图演示页 |

### 第 2 页：业务背景与痛点

标题：业务背景与痛点  
核心表达：法律辅助任务的难点不只是“生成文本”，而是“依据可追溯、过程可复核”。

页面线框：

```text
┌────────────────────────────────────────────────────────────┐
│ 标题区：业务背景与痛点                                      │
├──────────────────────────────┬─────────────────────────────┤
│ 左侧 45%：传统工作方式         │ 右侧 55%：LexHub 工作方式     │
│ 材料分散                      │ 工作台统一接入               │
│ 法规检索分离                  │ 法规 RAG 与来源摘要           │
│ 过程不可见                    │ DAG + Replay + Audit Log     │
│ 文书整理耗时                  │ 结构化报告与导出              │
├──────────────────────────────┴─────────────────────────────┤
│ 底部强调语：法律场景需要“事实、依据、过程、交付”的完整链路     │
└────────────────────────────────────────────────────────────┘
```

页面内容：

| 痛点 | 传统方式 | LexHub 解决方式 |
|---|---|---|
| 材料分散 | 文件、聊天记录、法规检索结果分离 | 工作台统一接入任务与材料 |
| 依据难追溯 | 法规、案例、观点散落在不同来源 | 法规 RAG、来源摘要和引用记录 |
| 过程不可见 | 只看到最终文本，难以复盘 | DAG、工具轨迹、Replay 回放 |
| 文书整理耗时 | 人工整合事实、依据和模板 | 结构化报告与 DOCX/PDF 导出 |
| 结论需复核 | 依赖人工逐项检查 | 交叉审查、审计链和人工复核入口 |

### 第 4 页：方案总览

标题：方案总览  
核心表达：LexHub 将法律任务组织为“材料、依据、流程、交付、复核”的闭环。

推荐图表：`assets/fig-1-2-lexhub-value-loop.svg`

页面线框：

```text
┌────────────────────────────────────────────────────────────┐
│ 标题区：方案总览                                            │
├──────────────────────────────────────┬─────────────────────┤
│ 左侧 70%：价值闭环图                  │ 右侧 30%：三条结论   │
│ [fig-1-2-lexhub-value-loop.svg]       │ 1. 不是单点问答      │
│                                      │ 2. 是任务办理链路    │
│                                      │ 3. 可运行可复核      │
├──────────────────────────────────────┴─────────────────────┤
│ 底部：输入材料 -> DAG 多智能体 -> 法律工具 -> 结构化成果 -> 审计复核 │
└────────────────────────────────────────────────────────────┘
```

讲解要点：

- LexHub 的核心价值不是让模型直接给出法律结论，而是把法律任务办理过程结构化。
- 材料、法规、类案、模板、文书和审计记录在同一个任务链路中流转。
- 结果页不是孤立文本，而是带有依据、风险和过程记录的交付件。

### 第 6 页：Co-Sight 改进对比

标题：Co-Sight 改进对比  
核心表达：LexHub 不是对原生 Co-Sight 的界面改名，而是在通用智能体底座上完成法律业务化封装。

页面线框：

```text

│ 标题区：Co-Sight 改进对比                                   │

│ 通用 Agent / Tool / Replay     │ 法律智能体 / 法规 RAG / 文书导出│
│ 通用执行日志                   │ DAG 执行视图 + 审计链 + 复核入口│
├──────────────────────────────┴─────────────────────────────┤
│ 底部：保留底座能力，新增法律工作流、知识库、业务页面和可信交付  │
```

页面内容：

| 对比维度 | 原生 Co-Sight | 律枢 LexHub 改进 |
|---|---|---|
| 产品定位 | 通用超级智能体框架 | 面向法律业务的智能服务平台 |
| 任务流程 | 通用 DAG 与执行链路 | 法律任务 DAG、条件触发、导出前复核 |
| 智能体角色 | 通用 Planner / Actor | 任务理解、证据质检、法规研究、文书生成、交叉审查、合规监测 |
| 工具体系 | 搜索、文件、代码执行等通用工具 | 法规 RAG、法律检索、材料库、文书导出、审计日志 |

包装重点：

- “继承”讲清楚：Co-Sight 提供多智能体、DAG、工具调用、Replay 等底座能力。
- “改造”讲清楚：团队完成法律业务页面、法律智能体配置、工作流、知识库、导出和审计链路。
- “价值”讲清楚：从通用执行框架变成可面向法律任务交付的行业应用。
