# 律枢 LexHub 国赛答辩 PPT 外包制作方案

项目名称：律枢 LexHub：基于 Co-Sight 的法律行业超级智能体工作台  
团队名称：六百六十六  
适用对象：PPT 外包设计师、团队内排版同学、后续答辩材料整理人员  
制作范围：封面和封底已由底板提供，本文只设计正文页。  
核心口径：LexHub 不是普通法律问答工具，而是面向法律业务的“用户端 + 管理端 + Co-Sight 执行引擎 + 法律知识工具 + 可信交付链”的双端超级智能体工作台。

## 第一章 产品缘起

### 第 1 页：需求来源

主标题：需求来源  
副标题：法律业务智能化的核心，不是“生成一段话”，而是处理材料、依据、流程和交付。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 01 需求来源                                      LexHub 标识 │
├────────────────────────────────────────────────────────────┤
│ 左侧 30%：设计动因                                            │
│  大号短句：法律任务不是一次问答                                │
│  三行说明：多材料 / 多依据 / 多交付物                           │
│                                                            │
│ 右侧 70%：法律业务处理链路                                     │
│  材料接收 → 事实梳理 → 法规检索 → 风险判断 → 文书生成 → 复核交付 │
│     │          │          │          │          │          │     │
│   材料散      耗时长      难追溯      主观性强    格式不一    难复盘 │
├────────────────────────────────────────────────────────────┤
│ 底部结论条：LexHub 的目标是把法律任务变成可编排、可追溯、可复核的执行流程 │
└────────────────────────────────────────────────────────────┘
```

推荐图形：


| 元素  | 画法                 |
| --- | ------------------ |
| 主链路 | 6 个节点横向排列，用细箭头连接   |
| 痛点  | 每个节点下方放 1 个红/金色小标签 |
| 结论条 | 深蓝底白字，横跨底部         |


页面文字：

- 法律业务往往不是单轮问答，而是围绕材料、法规、风险和文书展开的连续任务。
- 传统工作依赖人工拆解、人工检索、人工整理，效率和可复核性都受到限制。
- LexHub 面向这一类复杂任务，提供场景入口、任务编排、知识增强、过程回放和结果交付。

答辩话术：

我们做 LexHub 的出发点，是法律任务本身具有链路长、材料多、依据要求高的特点。真实工作不是问模型一个问题就结束，而是要接收材料、梳理事实、检索法规、判断风险、生成文书，并且最后还能复核。LexHub 想解决的就是这个链路问题。

### 第 2 页：用户场景

主标题：用户场景  
副标题：围绕不同法律工作者，把任务入口和交付物结构化。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 02 用户场景                                                 │
├──────────────────────┬─────────────────────────────────────┤
│ 左侧 28%：目标用户     │ 右侧 72%：场景 - 任务 - 交付物矩阵      │
│ 个人律师              │ 合同审查     → 条款风险清单 / 审查报告   │
│ 律所团队              │ 争议解决     → 事实时间线 / 证据目录     │
│ 企业法务              │ 公司事务     → 治理风险 / 决议草案       │
│ 合规人员              │ 合规知产     → 合规风险分级 / 整改建议   │
│ 法律学习者            │ 法规研究     → 法规摘要 / 研究报告       │
├──────────────────────┴─────────────────────────────────────┤
│ 底部 9 个场景标签：通用分析 / 材料解读 / 调研摘要 / 合同审查 / 公司事务 / 争议解决 / 合规知产 / 法规研究 / 文书起草 │
└────────────────────────────────────────────────────────────┘
```

推荐图表：


| 图表      | 数据                   |
| ------- | -------------------- |
| 用户-场景矩阵 | 5 类用户、5 个重点场景、对应交付物  |
| 底部场景标签  | 9 类场景，不要做太大，作为能力覆盖范围 |


答辩话术：

LexHub 面向的用户不只是一类。个人律师关注办案效率，律所团队关注流程统一，企业法务关注内控和审计，合规人员关注风险识别，法律学习者关注法规研究和文书训练。我们把这些需求整理成场景入口和交付物，而不是让用户从空白对话开始。

### 第 4 页：双端架构

主标题：双端架构  
副标题：用户端负责办任务，管理端负责管能力。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 04 双端架构                                                 │
├─────────────────────────┬──────────────────────────────────┤
│ 左侧：用户端 User Portal │ 右侧：管理端 Admin Console         │
│ 1 场景选择               │ 1 模型 / API 配置                  │
│ 2 任务描述               │ 2 法律知识库维护                   │
│ 3 材料上传               │ 3 工作流与策略规则                 │
│ 4 DAG 执行跟踪           │ 4 智能体与工具目录                 │
│ 5 结果查看与导出         │ 5 用户与会员标注                   │
├─────────────────────────┴──────────────────────────────────┤
│ 底部中台横梁：FastAPI 后端 + Co-Sight 执行引擎 + Legal KB + Replay / Audit │
└────────────────────────────────────────────────────────────┘
```

推荐视觉：

- 左侧用青蓝，代表业务入口；右侧用深蓝，代表配置管理。
- 底部中台横梁用深蓝长条，连接两端。
- 可在底部横梁上加 4 个小图标：API、Agent、Knowledge、Audit。

答辩话术：

我们把系统分成用户端和管理端。用户端面向业务人员，负责场景选择、任务输入、材料上传、执行跟踪和结果交付；管理端面向系统维护人员，负责模型、API、知识库、工作流、策略和会员标注。两端共用同一个 Co-Sight 执行中台。

### 第 5 页：Co-Sight 改造

主标题：底座改造  
副标题：从通用智能体框架，到法律行业双端应用系统。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 05 底座改造                                                 │
├───────────────┬───────────────────────┬────────────────────┤
│ 原生 Co-Sight  │ LexHub 行业化改造       │ 最终系统形态          │
│ 通用 Planner   │ 法律场景入口            │ 用户端工作台          │
│ Actor / Tool   │ 法律 DAG 工作流         │ 管理端控制台          │
│ Replay 机制    │ 法规知识工具            │ 可回放法律交付系统     │
├───────────────┴───────────────────────┴────────────────────┤
│ 下方 5 条改造路径：场景化 / 工作流化 / 知识增强 / 双端产品化 / 可信交付 │
└────────────────────────────────────────────────────────────┘
```

对比内容：


| 维度   | 原生 Co-Sight        | LexHub 改造          |
| ---- | ------------------ | ------------------ |
| 应用场景 | 通用任务执行             | 法律场景任务工作台          |
| 流程编排 | 通用 Planner / Actor | 法律 DAG + 条件触发      |
| 工具能力 | 搜索、代码等通用工具         | 法规检索、法律知识库、文书导出    |
| 过程展示 | replay 能力          | 执行页、结果页、回放页、审计说明   |
| 产品形态 | 技术底座               | 用户端 + 管理端 + 落地应用方案 |


答辩话术：

我们没有简单套用 Co-Sight，而是在它的通用智能体能力上做法律行业化改造。改造重点包括法律场景入口、法律 DAG 工作流、法规知识工具、用户端和管理端产品化，以及可回放、可审计的法律交付链路。

## 第三章 核心技术

### 第 7 页：智能体协同

主标题：智能体协同  
副标题：用角色分工承接真实法律业务流程。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 07 智能体协同                                               │
├────────────────────────────────────────────────────────────┤
│ 中央：任务理解智能体                                         │
│ 外圈 5 个智能体：证据质检 / 法规研究 / 文书生成 / 交叉审查 / 合规监测 │
│ 右侧输出：事实摘要 / 法规依据 / 风险清单 / 文书草稿 / 审计记录 │
├────────────────────────────────────────────────────────────┤
│ 底部指标条：6 类智能体 + 5 类工具能力 + 导出前复核             │
└────────────────────────────────────────────────────────────┘
```

智能体内容：


| 智能体     | 职责           | 对应产物 |
| ------- | ------------ | ---- |
| 任务理解智能体 | 识别目标、拆解 Plan | 任务计划 |
| 证据质检智能体 | 检查材料完整度      | 缺口清单 |
| 法规研究智能体 | 检索法规、案例、模板   | 依据摘要 |
| 文书生成智能体 | 生成报告、函件、合同草稿 | 文书初稿 |
| 交叉审查智能体 | 复核高风险内容      | 审查意见 |
| 合规监测智能体 | 导出前合规检查和审计记录 | 审计链  |


答辩话术：

法律任务需要分工协作，所以我们设计了六类法律智能体。任务理解负责拆解目标，证据质检检查材料，法规研究补充依据，文书生成形成交付物，交叉审查和合规监测负责导出前复核。这样系统的执行过程更接近真实法律工作。

### 第 8 页：DAG 编排

主标题：DAG 编排  
副标题：任务按状态动态流转，不是固定流水线。

版式框：

```text
┌────────────────────────────────────────────────────────────┐
│ 08 DAG 编排                                                 │
├────────────────────────────────────────────────────────────┤
│ 主路径：任务输入 → 任务理解 → 法规研究 → 文书生成 → 结果交付   │
│ 分支 1：材料不足 → 证据质检 → 补充材料                         │
│ 分支 2：引用不足 → 法规研究 → 来源摘要                         │
│ 分支 3：高风险 / 导出前 → 交叉审查 → 合规监测                   │
├────────────────────────────────────────────────────────────┤
│ 右侧工具栏：联网搜索 / 法规检索 / 文档处理 / 代码执行 / DOCX-PDF 导出 │
└────────────────────────────────────────────────────────────┘
```

推荐图形：

- 中间画主流程，用深蓝节点。
- 条件分支用青蓝虚线或金色折线。

答辩话术：

LexHub 的流程不是固定顺序。系统会根据任务状态触发不同节点：材料不足时进入证据质检，引用不足时进入法规研究，高风险或导出前进入交叉审查和合规监测。DAG 让任务执行更灵活，也让每一步为什么发生可以被解释。
