# Phase 02 - 设计原则与产品信息架构

本阶段把 Phase 01 的参考体系收束成 LexHub 自己的产品结构。重点不是视觉，而是明确用户为什么来、从哪里开始、在哪看到结果、管理端如何配置系统。

## 阶段目标

- 确定 LexHub 的产品定位和设计原则。
- 梳理用户端与管理端信息架构。
- 定义核心路径、页面任务和导航优先级。
- 为 Phase 03 Figma Design System 和 Phase 04 Key Screens 提供页面框架。

## 产品定位

LexHub 是法律业务智能工作台，不是普通 AI Chat，也不是营销型 SaaS。

核心定位：

> 面向真实法律业务的智能事项办理平台，提供事项受理、证据整理、法条检索、智能体协作、审查结论、文书输出和审计回放。

一句话产品心智：

> 把一件法律事项从材料、事实、依据到结论和文书，交给可追溯的智能体流程办理。

## 设计原则

### 1. 事项优先

用户不是来“聊天”的，而是来办理一件法律事项。界面应围绕事项生命周期组织，而不是围绕输入框组织。

落地要求：

- 主入口叫“事项受理”或“新建事项”。
- Workspace 首页展示场景、事实、材料、文书要素。
- 历史、材料、报告、审查都要能回到具体事项。

### 2. 过程透明

AI 不能只显示“生成中”。用户需要知道当前是谁在处理、用了什么材料、检索了什么依据、产出了什么中间结果。

落地要求：

- Run 页面展示智能体执行链。
- 工具调用、证据引用、法条检索有可展开详情。
- 状态包含 waiting、running、completed、warning、failed。

### 3. 依据可追溯

法律产品的可信度来自证据、法条、来源和审计链，而不是漂亮卡片。

落地要求：

- 结论旁边必须能看到依据。
- 报告页需要 Citation Block。
- 审查页需要 Evidence / Citation / Risk / Audit 并列入口。

### 4. 信息密度分层

LexHub 是工作台，不是宣传页。信息可以密，但必须分层。

落地要求：

- 默认展示摘要。
- 细节进入 Tabs、Drawer、展开区、详情面板。
- 管理端优先使用表格、列表、状态条、表单分组。

### 5. 后台可靠

管理端的任务是配置、监控和排错。它应该像控制台，不像营销展示页。

落地要求：

- Admin Overview 一眼看系统健康。
- Knowledge、Models、Connections、Policies 以配置和状态为主。
- 危险操作有确认，保存和测试有反馈。

### 6. 视觉克制

法律业务需要可信、安静、稳定。视觉强调应服务信息判断。

落地要求：

- 深法律蓝作为主品牌色。
- 金色只做强调，不做大面积背景。
- 减少大渐变、发光卡片、装饰性插画。
- 组件圆角、阴影、动效保持克制。

## 核心用户路径

### 用户端主路径

1. 登录
2. 新建事项
3. 选择业务场景
4. 补充事实和文书要素
5. 上传或选择材料
6. 启动智能体办理
7. 查看办理过程
8. 查看审查结论
9. 导出报告或文书
10. 回放归档

### 管理端主路径

1. 登录管理员账号
2. 查看系统总览
3. 检查模型、API、知识库状态
4. 配置模型和连接
5. 管理知识库导入与同步
6. 配置路由和审查策略
7. 查看用户和运行状态
8. 排查错误和审计记录

## 页面任务定义

| 页面 | 用户问题 | 页面主任务 | 主操作 |
| --- | --- | --- | --- |
| Landing | 这是什么产品？ | 建立产品定位与信任 | 登录 / 开始体验 |
| Login | 我如何进入？ | 选择身份并登录 | 登录 |
| Workspace Intake | 我要办理什么？ | 收集场景、事实、材料、要素 | 启动办理 |
| Workspace Run | 系统正在做什么？ | 展示智能体执行过程 | 查看结果 / 停止 / 重试 |
| Workspace Result | 结论是什么，可信吗？ | 展示报告、依据、证据、风险 | 导出文书 / 查看回放 |
| Review | 哪些结论需要复核？ | 人工审查和风险判断 | 通过 / 标记复核 |
| Materials | 我的材料在哪里？ | 管理材料、摘要、关联事项 | 上传 / 查看详情 |
| Evidence | 材料能否作为证据？ | 证据质量检查和整理 | 带入事项 |
| Reports | 有哪些已生成报告？ | 管理和导出历史报告 | 导出 / 查看 |
| Replay | 之前如何办理的？ | 回放事项处理链路 | 打开归档 |
| Admin Overview | 系统是否正常？ | 查看健康状态 | 测试 / 同步 / 查看详情 |
| Admin Knowledge | 知识库是否可用？ | 管理法规、案例、模板和导入 | 同步 / 导入 |
| Admin Connections | 能力连接是否正常？ | 管理模型、API、工具连接 | 测试连接 / 保存 |
| Admin Policies | 路由和审查规则是什么？ | 配置规则和风险策略 | 保存策略 |
| Admin Users | 谁在使用系统？ | 管理用户和权限 | 调整权限 |

## 导航策略

### 用户端导航

第一优先级只保留闭环路径：

- 事项受理
- 办理进度
- 审查结论
- 材料库
- 案件归档

第二优先级可进入更多业务能力：

- 报告
- 文书
- 证据整理
- 法规研究
- 智能体

账户类入口不要挤占主导航：

- 会员中心
- 用户档案

### 管理端导航

第一优先级：

- 系统概览
- 能力总览
- 知识库
- 策略规则
- 用户管理

第二优先级藏在页面 tabs 或详情内：

- 模型配置
- API 配置
- 路由规则
- 审查规则
- 分析图表
- 智能体详情

## 页面布局层级

所有应用页建议保持同一结构：

1. App Shell：侧边导航、用户身份、全局操作。
2. Page Header：页面标题、说明、状态、主操作。
3. Main Work Area：页面主任务。
4. Context Panel：证据、依据、智能体、历史、风险等上下文。
5. Feedback Layer：Toast、Dialog、Drawer、Loading、Empty、Error。
