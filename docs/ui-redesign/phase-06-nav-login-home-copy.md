# Phase 06 - 导航 / 登录 / 首页统一优化方案

## 目标

本方案用于统一 LexHub 的首页、营销导航、登录页、工作台导航表述。重点不是单独改几个按钮，而是建立一致的信息层级：

- 首页负责说明 LexHub 是什么、适合谁、从哪里开始。
- 登录页负责让用户清楚选择用户端或管理端。
- 工作台导航负责表达办理闭环，而不是堆页面名。
- 全部文案统一到“法律事项 / 办理路径 / 审查结论 / 文书交付 / 归档复核”的产品语言。

参考方向：

- Figma / SaaS landing 的短导航、短 CTA、强标题。
- Untitled UI / Framer SaaS kit 的清晰 section 命名。
- LexHub 自身法律办理链路，不套通用 AI 工具口径。

## 当前问题

### 1. 首页导航偏功能罗列

当前首页导航：

```text
首页 / 能力 / 办理流程 / 场景 / 会员方案
```

问题：

- “能力”偏抽象，像内部功能分类。
- “场景”和“办理流程”都对，但缺少“产品入口”的感觉。
- 和登录页、工作台的“事项受理 / 办理进度 / 审查结论”没有完全接上。

### 2. 登录页表达偏说明文

当前登录页标题：

```text
选择入口，进入律枢。
```

当前正文：

```text
用户端负责法律事项受理、材料提交与结论跟进，管理端负责知识库、模型能力与权限配置，共同构成完整办理闭环。
```

问题：

- 标题可用，但品牌感弱。
- 正文偏长，首屏读起来像说明文。
- Figma/SaaS 风格通常是短标题 + 两个明确入口 + 三步说明。

### 3. 工作台导航已有闭环，但可以更清晰

当前工作台导航：

```text
我的工作
- 事项受理
- 办理进度
- 审查结论
- 案件归档
- 材料库
```

这个方向基本正确。建议只微调，不大改：

- “我的工作”可改为“事项办理”。
- “材料库”可保留，也可改为“材料管理”。
- “案件归档”保留，法律语境清楚。

## 统一产品词典

建议全站采用以下主词：

| 概念 | 推荐词 | 避免词 |
| --- | --- | --- |
| 用户提交的业务 | 法律事项 / 事项 | 任务 |
| 系统处理过程 | 办理 / 推进 | 执行 |
| 处理路径 | 办理路径 / Matter Path | DAG、调度 |
| 输出报告 | 审查结论 / 审查报告 | 任务结果 |
| 文件生成 | 文书交付 | 产物生成 |
| 历史记录 | 案件归档 / 办理记录 | 回放 |
| 后台 | 管理控制台 | Admin |

说明：

- 面向用户页面尽量不用 DAG、Agent、LLM、API 等工程词。
- 管理后台可以保留模型、API、知识库等配置词。
- “智能体”可在解释办理链路时出现，但不作为首屏主标题。

## 首页导航方案

### 推荐导航

```text
产品
解决方案
办理流程
会员方案
```

右侧：

```text
登录
发起事项
```

理由：

- “产品”承接首页 Hero。
- “解决方案”承接合同审查、争议解决、公司事务。
- “办理流程”承接事项路径。
- “会员方案”承接 pricing。

### 当前锚点映射

| 新导航 | 当前 section | 说明 |
| --- | --- | --- |
| 产品 | `#home` 或 `#capabilities` | 首屏和能力区合并理解为产品介绍。 |
| 解决方案 | `#scenes` | 当前场景区。 |
| 办理流程 | `#workflow` | 当前流程区。 |
| 会员方案 | `#pricing` | 当前会员方案区。 |

### 不建议

```text
核心能力 / 场景 / 定价方案
```

原因：

- “核心能力”偏内部。
- “定价方案”可以换成更柔和的“会员方案”。
- “场景”可以升级为“解决方案”，更像 SaaS landing。

## 首页 Hero 文案方案

### 推荐版本

```text
让法律事项
沿路径办理。
```

正文：

```text
从材料受理到依据检索、风险复核与文书交付，LexHub 把每一步沉淀为可追溯的办理记录。
```

CTA：

```text
发起事项
查看办理流程
```

场景标签：

```text
合同审查 / 争议解决 / 公司事务 / 文书交付
```

### 更稳重版本

```text
法律事项，
有序推进。
```

正文：

```text
LexHub 将材料、依据、风险与结论串联为清晰办理路径，帮助法律事项完成审查、交付与归档。
```

### 更品牌化版本

```text
律枢，
让事项有迹可循。
```

正文：

```text
用统一路径组织法律事项，从事实材料到审查结论，每一步都保留依据、记录和复核线索。
```

推荐采用第一版。原因：

- 直接表达产品特色。
- 和 logo 的“路径 / 中轴 / 圆环”一致。
- 比泛化的“智能办理链路”更稳。

## 登录页方案

### 登录页定位

登录页不应只是表单页，而是“选择入口”的网关：

- 用户端：处理法律事项。
- 管理端：维护系统能力。

### 左侧标题推荐

```text
选择入口，
进入律枢。
```

或更品牌化：

```text
进入律枢，
继续办理。
```

推荐第二版。原因：

- 动作更明确。
- 登录页通常是承接用户返回，不只是第一次选择入口。

### 左侧正文推荐

```text
用户端处理事项受理、材料提交、办理进度与审查结论；管理端维护知识库、模型接入和权限配置。
```

比当前版本更短，仍保留完整信息。

### 登录页角色卡

用户端：

```text
用户端
发起法律事项，跟进办理路径、审查结论与文书交付。
```

管理控制台：

```text
管理控制台
配置知识库、模型服务、处理能力和用户权限。
```

### 登录页流程说明

当前三步可保留，但建议改短：

```text
1. 提交事项
选择场景，补充事实与材料。

2. 生成路径
系统整理材料、依据、风险与阶段结论。

3. 交付归档
输出审查结论、文书和可追溯办理记录。
```

### 表单按钮文案

| 场景 | 推荐 |
| --- | --- |
| 用户登录 | 进入用户端 |
| 管理员登录 | 进入管理控制台 |
| 注册 | 创建用户端账号 |
| 切换注册 | 创建账号 |
| 切换登录 | 已有账号，去登录 |

## 工作台导航方案

### 推荐侧边栏

分组名：

```text
事项办理
```

导航项：

```text
事项受理
办理进度
审查结论
材料管理
案件归档
```

排序建议：

1. 事项受理
2. 办理进度
3. 审查结论
4. 材料管理
5. 案件归档

理由：

- 前三项是主路径。
- 材料管理是支撑能力。
- 案件归档是历史沉淀。

如果保守迁移，可以只改分组名：

```text
我的工作 -> 事项办理
```

其余保持不变。

### 顶部面包屑

保持当前路由标签即可：

```text
事项受理 / 办理进度 / 审查结论 / 材料库 / 案件归档
```

若导航项改为“材料管理”，路由标签也应同步。

## Figma 风格落点

参考 Figma/SaaS landing 的方式，不是复制页面，而是采用这些表达：

### 1. 导航短而清晰

不要把所有功能都放进导航。首页导航只放决策入口。

### 2. 首屏大标题短

标题 8-14 个汉字最合适。避免：

```text
让法律事项进入智能办理链路
```

更推荐：

```text
让法律事项
沿路径办理。
```

### 3. CTA 动词明确

用：

```text
发起事项
查看办理流程
```

少用：

```text
免费体验
进入系统
了解更多
```

### 4. 登录页降低解释成本

用户不需要读长段落才能理解两个入口。角色卡本身要说明差异。

## 施工优先级

### P0 - 只改文案和命名

涉及：

- `src/pages/LandingPage.tsx`
- `src/pages/LoginPage.tsx`
- `src/config/navigation.ts`
- `src/components/layout/MarketingHeader.tsx`

改动：

- 首页 nav 文案。
- 首页 hero 文案。
- 登录页标题、正文、角色卡。
- 工作台导航分组名。

风险低，推荐先做。

### P1 - 改结构和视觉

涉及：

- Landing Hero C 的完整结构。
- 登录页左侧视觉升级。
- MarketingHeader 与 Landing 顶部导航统一。

需要配合截图验收。

### P2 - 全站同步

涉及：

- Result / Run / Materials / Replay 页面中残留的“任务 / 执行 / 回放”文案。
- 后台页面中面向用户可见的措辞。

## 验收标准

- 首页、登录页、工作台导航使用同一套产品词。
- 用户能在 5 秒内理解：
  - LexHub 做什么。
  - 从哪里发起事项。
  - 用户端和管理端区别。
- 导航不超过 5 个主要项。
- CTA 不使用含糊词。
- 移动端导航和标题不挤压、不断字。
- 不改业务逻辑，不影响登录、注册、权限判断和路由。
