# 律枢 LexHub 技术方案（国赛版）

## 1. 业务场景分析

### 1.1 赛道定位

- **赛道**：赛道六开放创新
- **场景**：法律行业通用任务协同（合同审查、争议解决、合规研究、文书生成）
- **产品名**：律枢 LexHub

### 1.2 业务痛点

| 痛点 | 传统方式 | LexHub + Co-Sight |
|------|----------|-------------------|
| 任务拆解 | 律师人工拆步骤 | Co-Sight Plan + DAG 自动编排 |
| 多源材料 | 邮件/文件夹分散 | 统一上传 + 证据质检智能体 |
| 法规引用 | 人工检索、难追溯 | 法规研究智能体 + 工具链 |
| 过程复核 | 难复盘 | replay.json 全链路回放 |
| 交付 | 手工整理 | 真实执行记录导出 DOCX/PDF |

### 1.3 用户画像

- 个人律师 / 法务顾问 → 个人 Pro / Ultra（规划）
- 试用用户 / 评委 → 体验版
- 律所 / 企业法务 → 定制交付（规划）

## 2. 技术架构设计

```
用户端 (React)
  ├── 工作台 /workspace     ← Co-Sight WebSocket 执行
  ├── 智能体 /agents        ← 调度可视化
  ├── 回放 /replay          ← replay.json
  └── 导出 /documents       ← 真实执行快照

管理端 (React Admin)
  ├── 模型/API/知识库配置
  ├── 工作流只读预览
  └── 用户与会员标注

后端 (FastAPI)
  ├── Co-Sight 执行与 replay
  ├── Demo API（能力展示、导出、快照）
  └── 工具链状态

配置层
  ├── config/legal-workflow.json
  └── config/knowledge/legal-knowledge-seed.json
```

### 2.1 智能体角色分工

| 智能体 | 职责 | 触发条件 |
|--------|------|----------|
| 任务理解 | 场景识别、Plan 拆解 | 任务提交 |
| 证据质检 | 材料完整度、缺口清单 | 完整度不足 |
| 法规研究 | 法规/案例检索引用 | 引用不足 |
| 文书生成 | 报告/律师函草稿 | 目标产出明确 |
| 交叉审查 | 风险表述、复核点 | 高风险/导出前 |

### 2.2 DAG 流程

见 `config/legal-workflow.json`：
- **条件分支**：材料/引用/风险条件触发不同节点
- **并行**：多工具可同时调用（replay 中可见）
- **回溯**：replay 恢复历史执行；跳步/返工通过 Plan 更新体现

## 3. 工具集成方案

≥3 类工具/API（详见 tool-api-reference.md）：

1. 联网搜索（Tavily / Google）
2. 代码执行（execute_code）
3. 文档/OCR 与材料上传
4. 法规检索（RAG 集成位）
5. 文书导出（DOCX/PDF）

## 4. 可信安全设计（可选加分）

| 机制 | 实现 |
|------|------|
| 人工复核点 | 审查规则 + 结果面板复核提示 |
| 过程可审计 | replay.json 逐行事件 |
| 导出溯源 | 执行快照附录（workspace、工具、阶段） |
| 权限分离 | 用户端 / 管理端角色分离 |
| 数据本地化 | Demo 阶段 localStorage；生产可私有化部署 |

## 5. 性能评估（示范）

| 指标 | 传统人工 | Co-Sight | 提升 |
|------|----------|----------|------|
| 单任务时间 | ~45min | ~12min | 约 73% |
| 引用可追溯 | ~60% | ~88% | +28% |
| 过程可复核 | 分散 | 全链路回放 | 100% 归档 |

> 详见 `/board` 与 `/analytics` 性能对比面板；正式商用需补实测数据。
