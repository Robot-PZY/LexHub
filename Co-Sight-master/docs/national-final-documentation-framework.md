# 国赛提交文档设计方案框架

> 用途：指导最终提交材料撰写。本文依据《中兴捧月“星匠师”巧匠精英挑战赛-超级智能体开发大赛 国赛赛题》要求，并结合律枢 LexHub 当前项目内容设计。

## 1. 赛题要求提炼

国赛题目要求参赛作品依托 Co-Sight 超级智能体框架，聚焦行业场景，设计并构建一个解决实际业务问题的 AI 解决方案。对 LexHub 来说，建议定位为：

- 赛道：赛道六，开放创新。
- 场景：法律行业智能化。
- 主题：超级智能体驱动的法律任务协同与可信交付。
- 产品名：律枢 LexHub。
- 核心任务：材料接入、证据质检、法规研究、文书生成、交叉审查、审计归档。

国赛硬性/关键要求可归纳为：

| 要求类别 | 赛题要求 | LexHub 文档中必须呈现的内容 |
|---|---|---|
| 框架底座 | 使用 Co-Sight 开源版本 2.0 及以上 | 说明基于中兴 Co-Sight 二次开发，保留原生多智能体、DAG、工具调用、replay |
| 多智能体 | 至少 2 个以上智能体协作 | 展示 6 类法律智能体及调度规则 |
| DAG 编排 | 体现并发、分支、循环等流程控制 | 展示 `legal-workflow.json`、DAG 图、条件分支、导出前强制审查 |
| 工具/API | 至少调用 3 类以上工具/API | 展示搜索、法律 RAG、文档处理、代码执行、文书导出、审计日志 |
| 多跳推理 | 链路深度大于 3 跳 | 用法律任务链路说明 5-6 跳推理过程 |
| 性能评估 | 与传统方案或基线模型做对比 | 提供人工处理 vs LexHub 的时间、追溯率、复核率对比 |
| 创新性 | 智能体设计、任务分解、工具封装、知识融合等创新 | 提炼不超过 5 条创新点 |
| 商业社会价值 | 可复制、可扩展、节约成本、提高效率 | 说明个人律师、律所、企业法务、合规团队推广路径 |
| 提交材料 | 技术方案、可运行代码/模型、创新总结报告，演示视频选交 | 对应准备文档、配置、代码、视频脚本和清单 |

## 3. 技术方案文档框架

### 3.1 封面与摘要

建议标题：

《律枢 LexHub：基于中兴 Co-Sight 的法律业务智能服务平台技术方案》

摘要建议包含：

- 项目定位：基于中兴 Co-Sight 的法律业务智能服务平台。
- 目标场景：合同审查、争议解决、公司治理、数据合规、法规研究、文书起草。
- 核心能力：多智能体协同、DAG 动态编排、法规 RAG、文书导出、回放审计。
- 国赛对应：开放创新赛道，满足多智能体、DAG、工具/API、多跳推理、可信安全、可复制推广。

### 3.2 业务场景分析

应写清楚：

1. 行业痛点
   - 法律材料分散，事实梳理耗时。
   - 法规、案例、证据之间难以建立可追溯链路。
   - 文书生成依赖人工经验，过程难复盘。
   - AI 法律工具容易黑盒化，缺少审查与审计。

2. 用户画像
   - 个人律师：提升材料整理和初稿生成效率。
   - 律所团队：统一案件处理流程和文书质量。
   - 企业法务：提高合同审查、合规研究和归档效率。
   - 评委/演示用户：通过样例材料快速复现能力。

3. 价值量化
   - 单任务处理时间从约 45 分钟降到约 12 分钟。
   - 引用可追溯率从约 60% 提升到约 88%。
   - 过程归档从分散记录变为 replay 全链路留痕。

注意：如果没有正式实验数据，应明确写成“示范测试/演示基准”，不要包装成生产实测。

### 3.3 技术架构设计

建议按四层展开：

1. 用户交互层
   - React + Vite 前端。
   - 登录、工作台、任务执行、任务结果、材料库、归档回放、管理端。

2. 智能体编排层
   - Co-Sight Planner / Actor 基础能力。
   - `agent-registry.json` 定义法律智能体组织。
   - `legal-workflow.json` 定义 DAG 节点、边和触发条件。

3. 工具与知识层
   - 联网搜索。
   - 法规 RAG。
   - 文档上传/解析。
   - 代码执行。
   - 文书导出。
   - 审计日志。

4. 数据与可信层
   - replay.json。
   - execution snapshot。
   - audit log。
   - 本地 Chroma 知识库。
   - 管理端运行时配置。

建议配图：

```text
用户任务/材料
  -> 任务理解智能体
  -> 条件分支：证据质检 / 法规研究
  -> 文书生成智能体
  -> 交叉审查智能体
  -> 合规监测智能体
  -> 导出文书 + replay 审计归档
```

### 3.4 智能体角色分工

建议使用表格：

| 智能体 | 角色 | 输入 | 工具 | 输出 |
|---|---|---|---|---|
| 任务理解智能体 | 调度者 | 用户描述、场景、材料状态 | Plan / routing | 场景识别、任务拆解、DAG 建议 |
| 证据质检智能体 | 工作者 | 上传材料、任务目标 | file_read、parse_document、file_upload | 缺口清单、事实要点、证据支撑度 |
| 法规研究智能体 | 工作者 | 争议焦点、关键词 | legal_rag、tavily_search、search_google | 法规引用、类案线索、来源说明 |
| 文书生成智能体 | 工作者 | 事实、法规、模板 | file_write、execute_code、export | 报告、律师函、意见书草稿 |
| 交叉审查智能体 | 审查者 | 初稿、事实、引用 | credibility_analysis | 一致性检查、幻觉风险、复核项 |
| 合规监测智能体 | 审查者 | 导出内容、执行记录 | audit_log、export | 审计链、合规提示、导出门禁 |

### 3.5 DAG 任务编排说明

需要突出：

- 分支：材料完整度低于 70% 时触发证据质检。
- 分支：法规引用覆盖低于 55% 时触发法规研究。
- 条件：用户需要报告/律师函时触发文书生成。
- 强制节点：高风险或导出前触发交叉审查。
- 归档节点：导出前触发合规监测和审计链。
- 多跳深度：至少覆盖“理解 -> 质检 -> 研究 -> 生成 -> 审查 -> 归档”。

可引用文件：

- `config/legal-workflow.json`
- `config/agent-registry.json`
- `cosight_frontend/src/components/workspace/DagExecutionPanel.tsx`
- `cosight_frontend/src/components/workspace/DagGraphPanel.tsx`

### 3.6 工具/API 集成方案

建议按国赛“至少 3 类工具/API”写成 6 类：

| 工具类别 | 项目实现 | 价值 |
|---|---|---|
| 联网搜索 | Tavily / Google / Search API 集成位 | 补充公开资料与外部事实 |
| 法规检索 | `legal_search_toolkit.py`、法律 KB、Chroma | 提供法律依据和引用溯源 |
| 文档处理 | 上传、读取、解析、材料库 | 处理 PDF/DOCX/图片/表格等案件材料 |
| 代码执行 | Co-Sight 内置执行能力 | 支持结构化整理、统计、格式转换 |
| 文书导出 | DOCX/PDF 导出服务 | 形成可交付报告或文书 |
| 可信审计 | replay、audit log、execution snapshot | 支持复盘、追责、答辩展示 |

### 3.7 核心算法/策略说明

这里不需要强行写复杂模型算法，建议写“任务调度策略 + 证据/引用评分 + 审查策略”：

1. 任务拆解策略
   - 根据场景、描述关键词、材料状态、目标产出识别任务路径。
   - 将法律任务拆为事实、证据、依据、输出、审查、归档六类子任务。

2. 动态路由策略
   - 材料完整度、引用覆盖率、风险等级、是否需要输出共同决定智能体调度。
   - 对高风险/导出任务设置强制审查节点。

3. 知识融合策略
   - 本地法规库、外部法规 API、联网搜索、模板库混合检索。
   - 生成时优先引用可追溯来源。

4. 可信审查策略
   - 对事实、引用、结论、输出文书建立审查清单。
   - replay 记录工具调用和阶段结论。

### 3.8 可信安全设计

建议写成“当前已实现 + 后续生产化增强”：

当前已实现/演示：

- 用户端与管理端入口分离。
- 上传文件类型、数量、大小校验。
- replay 全链路过程记录。
- execution snapshot 支持导出溯源。
- audit log 支持审计查看。
- 导出前交叉审查与合规监测节点。

生产化增强：

- 服务端鉴权替换前端演示级鉴权。
- 租户隔离和角色权限。
- 敏感信息脱敏。
- 私有化部署。
- 审计日志不可篡改存储。

### 3.9 性能评估报告

建议包含：

| 指标 | 传统方式 | LexHub 演示基准 | 说明 |
|---|---|---|---|
| 单任务处理时间 | 约 45 分钟 | 约 12 分钟 | 从材料梳理到初稿输出 |
| 引用可追溯率 | 约 60% | 约 88% | 是否能定位到法规/案例/来源 |
| 过程复核 | 分散记录 | replay 全链路归档 | 可查看阶段、工具、输出 |
| 文书初稿形成 | 手工整理 | 自动生成并导出 | 需人工最终复核 |

建议补充测试说明：

- 测试环境：本地 Windows、Python、Node、DeepSeek 或等价 LLM。
- 测试材料：`test/case-01-labor-dispute`、`case-02-corporate-governance`、`case-03-data-compliance`。
- 测试方式：同一任务分别以人工流程和 LexHub 演示流程处理，统计时间和输出质量。

## 4. 创新总结报告框架

创新总结报告是必交材料，建议控制在 4-6 页或 1500-2500 字。结构如下：

### 4.1 方案概述

用一段话说明：

律枢 LexHub 是基于中兴 Co-Sight 的法律业务智能服务平台，通过多智能体协同和 DAG 动态编排，将法律任务拆解为材料接入、证据质检、法规研究、文书生成、交叉审查和审计归档，形成可运行、可复核、可扩展的法律 AI 解决方案。

### 4.2 创新点（不超过 5 条）

建议最终使用这 5 条：

1. 法律任务的状态驱动多智能体调度。
2. Co-Sight DAG 与法律业务条件深度结合。
3. 法规 RAG、外部检索、模板库的混合知识融合。
4. 基于 replay 的可信审计与执行快照导出。
5. 面向竞赛和行业交付的一体化工作台与管理端。

### 4.3 技术难点与解决方案

| 难点 | 解决方案 |
|---|---|
| 法律任务非线性，不能固定流水线 | 用材料完整度、引用覆盖率、风险等级驱动 DAG 分支 |
| AI 结论容易不可追溯 | 引入法规 RAG、来源记录、replay、审计日志 |
| 原生执行过程不易展示 | 构建 DAG 执行视图、工具轨迹、阶段结果面板 |
| 文书输出需要可交付 | 将 execution snapshot 转为 DOCX/PDF 导出内容 |
| 现场演示需要低门槛复现 | 提供一键启动脚本、测试材料、演示路线和提交清单 |

### 4.4 推广应用前景

建议按用户层级写：

- 个人律师：低成本获得材料整理、法规研究、文书初稿能力。
- 中小律所：统一办案流程和文书质量。
- 企业法务：批量合同审查、合规研究、归档审计。
- 行业扩展：同一 Co-Sight 底座可迁移到财税、审计、合规、保险理赔等强证据链场景。

### 4.5 团队协作经验

这里需要按你们真实分工补充。建议模板：

- 产品与赛题对齐：负责场景定位、国赛要求拆解、演示路线。
- 前端开发：负责 React 工作台、DAG 可视化、管理端、导出交互。
- 后端开发：负责 Co-Sight 接入、上传、知识库、文书导出、审计接口。
- 算法/智能体设计：负责智能体角色、DAG 规则、提示词与工具集成。
- 测试与材料：负责样例 PDF、测试用例、提交文档和视频脚本。

## 7. 文档撰写注意事项

1. 明确区分 Co-Sight 原生能力和团队二次开发能力。
2. 所有赛题硬性要求都要在文档中有对应位置，不要只在代码里体现。
3. 性能数据若未正式测试，要标注“演示基准”或“示范测试”。

6. 答辩材料中尽量引用具体文件路径，如 `legal-workflow.json`、`agent-registry.json`、`legal_search_toolkit.py`。
