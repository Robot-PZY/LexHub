# Phase 6：演示交付与可恢复体验

## 阶段目标

本阶段将已经可执行的多智能体流程整理为评委能够直接理解、复现和带走的演示闭环：任务可回放、阶段与工具调用可量化、异常可重试、最终结果可稳定导出。

## 关键改进

### 回放中心

- 回放列表由单纯显示事件数，扩展为阶段数、已完成/阻塞状态、智能体数、工具事件数和最终结果状态。
- 使用最后一次计划快照作为标题与场景依据，避免早期中间消息覆盖最终任务语义。
- 加载失败显示明确原因与重试入口，不再以空列表掩盖服务异常。

### 结果交付

- 预览采用模板模式，保证打开即有稳定内容。
- 正式导出采用 execution 模式，直接消费真实执行快照，不重复调用大模型，也不会在导出时改变结论。
- DOCX 与 PDF 使用同一份结构化章节数据生成，并保留 Co-Sight 执行来源。
- 导出状态以内联 `aria-live` 反馈呈现，替代阻断式浏览器弹窗。

### 展示口径

- “核验智能体”承担事实、依据、引用与输出一致性的自动校验。
- 不将人工复核设计为智能体、DAG 阶段或交付前置条件。
- 对无法完成的节点使用阻塞状态和降级结果，最终仍能形成可解释、可回放的交付物。

## 演示闭环

1. 用户提交法律任务与材料。
2. Planner 动态选择专业智能体并生成带依赖关系的 DAG。
3. 工作台按阶段展示智能体和工具调用顺序、运行状态及结构化结果。
4. Finalizer 汇总完成与阻塞节点，由核验智能体生成自动质量校验结果。
5. 回放中心复现完整轨迹，交付面板导出 DOCX/PDF。

## 验收

运行：

```powershell
python tools/validate_phase6_demo_delivery.py
```

脚本验证：

- DOCX/PDF 文件签名、大小、媒体类型和中文文件名；
- 至少一条真实回放同时包含最终结果、阶段与工具证据；
- 回放指标、错误恢复和导出模式的前后端契约；
- 核心交付界面不存在旧的人工复核前置措辞。
